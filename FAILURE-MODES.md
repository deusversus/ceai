# CE AI Suite — Failure Modes Analysis

**Date:** 2026-04-08
**Scope:** Full codebase review — security, stability, AI safety, test coverage, credential management

---

## Executive Summary

CE AI Suite is a Cheat Engine-class memory analysis tool with an integrated AI operator (MAF agent, ~90 tool functions). This document identifies failure modes across the critical attack surface areas and assesses existing mitigations.

**Key finding:** The project has substantially more test coverage than documented (570 test methods across 47 files vs. the "65 tests" stated in README), but coverage is unevenly distributed — AI operator infrastructure is well-tested while AI output validation and provider failover remain gaps.

---

## 1. AI Output Safety

**Risk Level:** HIGH

**Failure Mode:** The AI operator generates Auto Assembler scripts, memory write parameters, and breakpoint configurations. Malicious, hallucinated, or malformed AI outputs could corrupt game state, crash target processes, or execute unintended code.

### Existing Mitigations

| Mitigation | Location | Effectiveness |
|------------|----------|---------------|
| 12 dangerous tools require user approval | `DangerousToolsTests.cs` — `DangerousTools.Names` set | Strong — WriteMemory, SetBreakpoint, ToggleScript, etc. all gated |
| `PermissionEngine` with glob-matching rules | `AgentLoopTests.cs:410-449` — first-match-wins rule evaluation | Strong — tested with Allow/Deny/Ask effects |
| `HookRegistry` pre-tool blocklist | `AgentLoopTests.cs:454-464` — `ToolBlocklistHook` | Strong — can block specific tools before execution |
| `MaxToolCallsHook` limits total tool invocations | `AgentLoopTests.cs:467-479` — blocks after N calls | Moderate — prevents runaway loops |
| `ValidateScript` syntax checking | `AiToolFunctions.Scripts.cs:80-98` — parse validation | Moderate — catches syntax errors, not semantic issues |
| `ValidateScriptDeep` with live assert verification | `AiToolFunctions.Scripts.cs:103-200+` — reads live memory to verify `assert()` directives | Strong — verifies script byte patterns match live process |
| `ProbeTargetRisk` breakpoint risk assessment | `AiToolFunctions.Breakpoints.cs` — LOW/MEDIUM/HIGH/CRITICAL | Moderate — evaluates target address safety |

### Gaps

- **No fuzzing tests** for AI-generated assembly code — `ValidateScript` only validates syntax, not opcode safety
- **No adversarial output tests** — what happens if the AI returns malformed JSON, truncated scripts, or unexpected argument types?
- **No static analysis of generated bytecode** — code caves accept arbitrary assembled bytes without opcode-level validation
- **`DryRunHookInstall`** exists but has no dedicated tests

### Recommendations

1. Add test cases that feed malformed/adversarial script content to `ValidateScript` and `ValidateScriptDeep`
2. Add opcode allowlist or deny-dangerous-opcode check before code cave execution
3. Test `DryRunHookInstall` with edge cases (overlapping caves, invalid addresses)

## 2. AI Provider Switching & Resilience

**Risk Level:** HIGH

**Failure Mode:** The app supports 5 AI providers (GitHub Copilot, OpenAI, Anthropic, Gemini, OpenAI-compatible). Provider switching mid-operation, token expiration, rate limiting, or API errors could leave the agent in an inconsistent state.

### Existing Mitigations

| Mitigation | Location | Effectiveness |
|------------|----------|---------------|
| `ErrorClassifier` categorizes API errors | `AgentLoopTests.cs:42-100` — rate limit, overloaded, auth, prompt-too-long, retriable | Strong — comprehensive classification |
| `CopilotAuthPolicy` handles 401/403 with auto-retry | `ChatClientFactory.cs:162-188` — ForceRefresh on auth failures | Strong — automatic token rotation |
| `TokenBudget` tracks costs and enforces limits | `AgentLoopTests.cs:104-198` — cumulative tracking, exhaustion detection, budget fractions | Strong — prevents runaway costs |
| `ApiKeyValidator` validates keys per provider | `ApiKeyValidatorTests.cs` — empty, whitespace, unknown provider, missing endpoint | Moderate — validates format, not live status |
| `ChatClientFactory` abstracts provider creation | `ChatClientFactory.cs:45-67` — switch on provider string | Moderate — clean abstraction |
| Cancellation propagation tested | `AgentLoopTests.cs:599-615` — CancellationToken stops slow clients | Strong — emits proper Error event |

### Gaps

- **No failover tests** — if a provider returns repeated 500s mid-conversation, no test verifies graceful degradation
- **No provider-switch-mid-conversation test** — changing provider while the agent has active context is untested
- **Copilot token refresh race condition** — `ForceRefreshAsync` is called on 401/403 but concurrent requests could race
- **Gemini OAuth blocking call** — `CreateGeminiAuto` calls `.GetAwaiter().GetResult()` (`ChatClientFactory.cs:77`), which could deadlock on UI thread
- **No retry backoff tests** — `ErrorClassifier.ParseRetryAfter` is tested loosely (`retryAfter is null or { TotalSeconds: >= 0 }`)

### Recommendations

1. Add integration tests simulating provider failures (mock 500, 429, timeout sequences)
2. Test provider switching mid-conversation — verify chat history manager handles the transition
3. Make Gemini OAuth token retrieval async end-to-end (remove `.GetAwaiter().GetResult()`)
4. Add concurrency test for `CopilotTokenService.ForceRefreshAsync` under parallel requests

## 3. Process Stability & Crash Recovery

**Risk Level:** MEDIUM-HIGH

**Failure Mode:** Breakpoint installation, code cave hooks, and memory writes can crash or freeze the target process. If recovery fails, the target is left in a corrupted state.

### Existing Mitigations

| Mitigation | Location | Effectiveness |
|------------|----------|---------------|
| `ProcessWatchdogService` with multi-signal health checks | `ProcessWatchdogService.cs:267-350` — WM_NULL, ReadProcessMemory canary, CPU time progress (2-of-3 voting) | Strong — sophisticated responsiveness detection |
| Transactional install with auto-rollback | `ProcessWatchdogService.cs:87-189` — pre-check, install, verify, commit/rollback | Strong — PageGuard storm detection at 500ms and 750ms |
| Unsafe address registry | `ProcessWatchdogService.cs:39-81` — remembers address+mode combos that caused freezes | Strong — prevents repeat failures |
| Freeze telemetry logging | `ProcessWatchdogService.cs:225-258` — JSON freeze logs to `%LOCALAPPDATA%\CEAISuite\logs` | Moderate — forensic but not preventive |
| `EmergencyRestorePageProtection` and `ForceDetachAndCleanup` | `BreakpointContracts.cs`, `AiToolFunctions.Breakpoints.cs` | Strong — nuclear recovery options |
| `CrashRecoveryTests` for address table recovery | `CrashRecoveryTests.cs` — export/import recovery, corrupt file handling | Moderate — covers serialization recovery |
| Continuous monitoring via `WatchdogMonitor` | `ProcessWatchdogService.cs:424-531` — background heartbeat loop | Strong — ongoing protection post-install |

### Gaps

- **No stress tests** for concurrent breakpoints + rapid process state changes
- **No test for watchdog monitor rollback under real process conditions** — all tests use stubs
- **No deadlock detection** — `IsProcessResponsive` uses `Thread.Sleep(50)` for CPU sampling, which could miss certain deadlock patterns
- **No test for `WatchdogGuard` disposal race** — `Interlocked.Exchange` looks correct but is untested
- **CrashRecoveryTests** only test address table serialization, not actual breakpoint/hook crash scenarios

### Recommendations

1. Add stress tests installing/removing multiple breakpoints in rapid succession
2. Test `WatchdogGuard` disposal from multiple threads simultaneously
3. Consider adding thread-level deadlock detection (wait chain traversal via `WCT` API)

## 5. Credential & Secret Management

**Risk Level:** LOW-MEDIUM

**Failure Mode:** API keys for OpenAI, Anthropic, Gemini, and GitHub tokens are stored on disk. Compromise of the settings file or DPAPI keys could leak credentials.

### Existing Mitigations

| Mitigation | Location | Effectiveness |
|------------|----------|---------------|
| DPAPI encryption for all API keys | `AppSettingsService.cs` — `EncryptedApiKey`, `EncryptedAnthropicApiKey`, etc. | Strong — tied to Windows user login |
| Runtime plaintext marked `[JsonIgnore]` | `AppSettingsService.cs:17-60` — never serialized to disk | Strong — keys only in memory |
| Per-provider key separation | `AppSettings` class — separate encrypted fields per provider | Good — limits blast radius |
| Environment variable fallback | README — `OPENAI_API_KEY`, `ANTHROPIC_API_KEY` auto-detected | Moderate — standard practice |
| API key validation before use | `ApiKeyValidatorTests.cs` — empty, whitespace, format checks | Moderate |

### Gaps

- **No key rotation mechanism** — keys are stored until manually changed
- **No secure key deletion on uninstall** — `%LOCALAPPDATA%\CEAISuite\settings.json` persists
- **No memory protection** for runtime plaintext keys — `string` fields are not pinned/zeroed
- **Gemini OAuth credentials** (client ID, client secret) also stored via DPAPI but refresh tokens have no expiry tracking

### Recommendations

1. Consider `SecureString` or pinned memory for runtime key storage (though .NET's SecureString is deprecated — evaluate alternatives)
2. Add uninstall cleanup guidance or script
3. Track and warn on Gemini refresh token age

**Risk Level:** LOW (for current scope)

### Existing Mitigations

- `StubEngineFacade` enables testing without Windows process APIs

### Assessment

Well-documented and acknowledged. Not a concern for the current Windows-only v1.

### Existing Mitigations

- `ProbeTargetRisk` evaluates target address safety before breakpoint installation

### Gaps

- No real-world anti-cheat testing (inherently difficult to automate)
- No detection simulation tests

## 8. Cheat Table Serialization

**Risk Level:** LOW

**Failure Mode:** Loading/saving `.CT` cheat tables could corrupt data, lose pointer chains, or misinterpret data types.

### Existing Mitigations

- 10 comprehensive round-trip tests in `CheatTableRoundTripTests.cs`:
  - Simple entry address/type preservation
  - Script entry content preservation
  - Nested group hierarchy preservation
  - Pointer entry with offsets
  - Dropdown list values
  - ShowAsHex/ShowAsSigned flags
  - Lua script preservation
  - Empty CT handling
  - Mixed data types (Int32, Float, Double, String, ByteArray)
  - Entry count verification after round-trip
- `CrashRecoveryTests.cs` verifies export/import including corrupt file rejection

### Gaps

- **No malicious CT file test** — what happens with XML entities, XXE attacks, extremely deep nesting?
- **No very large CT file test** — performance with thousands of entries is untested

### Recommendation

Add a test with a crafted adversarial `.CT` file (XXE, entity expansion, >1000 entries).

## 9. Lua Scripting Engine

**Risk Level:** MEDIUM

**Failure Mode:** Lua scripting is partially implemented (~100 tests across 5 Lua test files) but represents approximately 15% of Cheat Engine's full Lua API surface. Scripts relying on CE-specific Lua functions may silently fail or behave unexpectedly.

### Existing Mitigations

- `LuaEngineTests.cs` (34 tests), `LuaCeApiTests.cs` (26 tests), `LuaGapClosureTests.cs` (13 tests), `LuaAutoAssemblerIntegrationTests.cs` (14 tests), `LuaConsoleViewModelTests.cs` (13 tests)
- CE API bindings are stub-tested for the implemented surface area

### Gaps

- **No Lua sandbox escape tests** — if Lua `os.execute` or `io.popen` are accessible, scripts could execute arbitrary system commands
- **Incomplete CE Lua API** — users porting CE Lua scripts will hit missing functions
- **No test for Lua memory limits** — infinite loops or large allocations could hang the app

### Recommendation

1. Verify Lua sandbox restricts `os`, `io`, `debug`, `loadfile` modules
2. Add memory/CPU time limits for Lua script execution
3. Document which CE Lua functions are supported vs. missing

## 10. Tool-to-UI Parity

**Risk Level:** LOW-MEDIUM

**Failure Mode:** Only ~21% of the ~90 AI tool functions have manual UI equivalents. Users relying on the UI without AI cannot access most capabilities.

### Assessment

This is a design decision, not a bug — the philosophy is "AI assists, not replaces." However, if the AI provider is unavailable, users lose access to ~70 functions including pointer scanning, structure dissection, AOB signature generation, and session management.

### Recommendation

Prioritize UI for critical-path functions: pointer scanner, memory scanner refinement, and session save/load.
