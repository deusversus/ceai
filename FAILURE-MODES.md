# CE AI Suite ‚Äî Failure Modes Analysis

**Date:** 2026-04-08
**Scope:** Full codebase review ‚Äî security, stability, AI safety, test coverage, credential management

---

## Executive Summary

CE AI Suite is a Cheat Engine-class memory analysis tool with an integrated AI operator (MAF agent, ~90 tool functions). This document identifies failure modes across the critical attack surface areas and assesses existing mitigations.

**Key finding:** The project has substantially more test coverage than documented (570 test methods across 47 files vs. the "65 tests" stated in README), but coverage is unevenly distributed ‚Äî AI operator infrastructure is well-tested while AI output validation and provider failover remain gaps.

---

## 1. AI Output Safety

**Risk Level:** HIGH

**Failure Mode:** The AI operator generates Auto Assembler scripts, memory write parameters, and breakpoint configurations. Malicious, hallucinated, or malformed AI outputs could corrupt game state, crash target processes, or execute unintended code.

### Existing Mitigations

| Mitigation | Location | Effectiveness |
|------------|----------|---------------|
| 12 dangerous tools require user approval | `DangerousToolsTests.cs` ‚Äî `DangerousTools.Names` set | Strong ‚Äî WriteMemory, SetBreakpoint, ToggleScript, etc. all gated |
| `PermissionEngine` with glob-matching rules | `AgentLoopTests.cs:410-449` ‚Äî first-match-wins rule evaluation | Strong ‚Äî tested with Allow/Deny/Ask effects |
| `HookRegistry` pre-tool blocklist | `AgentLoopTests.cs:454-464` ‚Äî `ToolBlocklistHook` | Strong ‚Äî can block specific tools before execution |
| `MaxToolCallsHook` limits total tool invocations | `AgentLoopTests.cs:467-479` ‚Äî blocks after N calls | Moderate ‚Äî prevents runaway loops |
| `ValidateScript` syntax checking | `AiToolFunctions.Scripts.cs:80-98` ‚Äî parse validation | Moderate ‚Äî catches syntax errors, not semantic issues |
| `ValidateScriptDeep` with live assert verification | `AiToolFunctions.Scripts.cs:103-200+` ‚Äî reads live memory to verify `assert()` directives | Strong ‚Äî verifies script byte patterns match live process |
| `ProbeTargetRisk` breakpoint risk assessment | `AiToolFunctions.Breakpoints.cs` ‚Äî LOW/MEDIUM/HIGH/CRITICAL | Moderate ‚Äî evaluates target address safety |

### Gaps

- **No fuzzing tests** for AI-generated assembly code ‚Äî `ValidateScript` only validates syntax, not opcode safety
- **No adversarial output tests** ‚Äî what happens if the AI returns malformed JSON, truncated scripts, or unexpected argument types?
- **No static analysis of generated bytecode** ‚Äî code caves accept arbitrary assembled bytes without opcode-level validation
- **`DryRunHookInstall`** exists but has no dedicated tests

### Recommendations

1. Add test cases that feed malformed/adversarial script content to `ValidateScript` and `ValidateScriptDeep`
2. Add opcode allowlist or deny-dangerous-opcode check before code cave execution
3. Test `DryRunHookInstall` with edge cases (overlapping caves, invalid addresses)

## 2. AI Provider Switching & Resilience

**Risk Level:** HIGH

**Failure Mode:** The app supports 5 AI providers (GitHub Copilot, OpenAI, Anthropic, Gemini, OpenAI-compatible). Provider switching mid-operation, token expiration, rate limiting, or API errors could leave the agent in an inconsistent state.

### Existing Mitigations

| Mitigation | Location | Effectiveness |
|------------|----------|---------------|
| `ErrorClassifier` categorizes API errors | `AgentLoopTests.cs:42-100` ‚Äî rate limit, overloaded, auth, prompt-too-long, retriable | Strong ‚Äî comprehensive classification |
| `CopilotAuthPolicy` handles 401/403 with auto-retry | `ChatClientFactory.cs:162-188` ‚Äî ForceRefresh on auth failures | Strong ‚Äî automatic token rotation |
| `TokenBudget` tracks costs and enforces limits | `AgentLoopTests.cs:104-198` ‚Äî cumulative tracking, exhaustion detection, budget fractions | Strong ‚Äî prevents runaway costs |
| `ApiKeyValidator` validates keys per provider | `ApiKeyValidatorTests.cs` ‚Äî empty, whitespace, unknown provider, missing endpoint | Moderate ‚Äî validates format, not live status |
| `ChatClientFactory` abstracts provider creation | `ChatClientFactory.cs:45-67` ‚Äî switch on provider string | Moderate ‚Äî clean abstraction |
| Cancellation propagation tested | `AgentLoopTests.cs:599-615` ‚Äî CancellationToken stops slow clients | Strong ‚Äî emits proper Error event |

### Additional Mitigations (found in deep inspection)

| Mitigation | Location | Effectiveness |
|------------|----------|---------------|
| `ModelSwitcher` with fallback chain | `AgentLoop/ModelSwitcher.cs` ‚Äî `FallbackToNext()` on 3 consecutive overloads | Strong ‚Äî automatic model downgrade |
| `RetryPolicy` with exponential backoff | `AgentLoop/RetryPolicy.cs` ‚Äî rate limit, overload, context-length-exceeded handling | Strong ‚Äî cascading backoff strategy |
| `TriggerCooldown` for rate-limited models | `ModelSwitcher.cs` ‚Äî temporarily switches to faster model, auto-restores | Good ‚Äî avoids expensive model during high load |
| `CompactionPipeline` for context overflow | `AgentLoop/CompactionPipeline.cs` + `PostCompactionRestorer.cs` | Strong ‚Äî summarizes history, restores essential tool results |

### Gaps

- **No failover chain test** ‚Äî `ModelSwitcher.FallbackToNext()` and `TriggerCooldown()` have no unit tests verifying the fallback chain behavior
- **No provider-switch-mid-conversation test** ‚Äî changing provider while the agent has active context is untested
- **Copilot token refresh race condition** ‚Äî `ForceRefreshAsync` is called on 401/403 but concurrent requests could race
- **Gemini OAuth blocking call** ‚Äî `CreateGeminiAuto` calls `.GetAwaiter().GetResult()` (`ChatClientFactory.cs:77`), which could deadlock on UI thread
- **No compaction round-trip test** ‚Äî `CompactionPipeline` and `PostCompactionRestorer` are mentioned but no tests verify data fidelity through compaction

### Recommendations

1. Add integration tests simulating provider failures (mock 500, 429, timeout sequences)
2. Test provider switching mid-conversation ‚Äî verify chat history manager handles the transition
3. Make Gemini OAuth token retrieval async end-to-end (remove `.GetAwaiter().GetResult()`)
4. Add concurrency test for `CopilotTokenService.ForceRefreshAsync` under parallel requests

## 3. Process Stability & Crash Recovery

**Risk Level:** MEDIUM-HIGH

**Failure Mode:** Breakpoint installation, code cave hooks, and memory writes can crash or freeze the target process. If recovery fails, the target is left in a corrupted state.

### Existing Mitigations

| Mitigation | Location | Effectiveness |
|------------|----------|---------------|
| `ProcessWatchdogService` with multi-signal health checks | `ProcessWatchdogService.cs:267-350` ‚Äî WM_NULL, ReadProcessMemory canary, CPU time progress (2-of-3 voting) | Strong ‚Äî sophisticated responsiveness detection |
| Transactional install with auto-rollback | `ProcessWatchdogService.cs:87-189` ‚Äî pre-check, install, verify, commit/rollback | Strong ‚Äî PageGuard storm detection at 500ms and 750ms |
| Unsafe address registry | `ProcessWatchdogService.cs:39-81` ‚Äî remembers address+mode combos that caused freezes | Strong ‚Äî prevents repeat failures |
| Freeze telemetry logging | `ProcessWatchdogService.cs:225-258` ‚Äî JSON freeze logs to `%LOCALAPPDATA%\CEAISuite\logs` | Moderate ‚Äî forensic but not preventive |
| `EmergencyRestorePageProtection` and `ForceDetachAndCleanup` | `BreakpointContracts.cs`, `AiToolFunctions.Breakpoints.cs` | Strong ‚Äî nuclear recovery options |
| `CrashRecoveryTests` for address table recovery | `CrashRecoveryTests.cs` ‚Äî export/import recovery, corrupt file handling | Moderate ‚Äî covers serialization recovery |
| Continuous monitoring via `WatchdogMonitor` | `ProcessWatchdogService.cs:424-531` ‚Äî background heartbeat loop | Strong ‚Äî ongoing protection post-install |

### Gaps

- **No stress tests** for concurrent breakpoints + rapid process state changes
- **No test for watchdog monitor rollback under real process conditions** ‚Äî all tests use stubs
- **No deadlock detection** ‚Äî `IsProcessResponsive` uses `Thread.Sleep(50)` for CPU sampling, which could miss certain deadlock patterns
- **No test for `WatchdogGuard` disposal race** ‚Äî `Interlocked.Exchange` looks correct but is untested
- **CrashRecoveryTests** only test address table serialization, not actual breakpoint/hook crash scenarios

### Recommendations

1. Add stress tests installing/removing multiple breakpoints in rapid succession
2. Test `WatchdogGuard` disposal from multiple threads simultaneously
3. Consider adding thread-level deadlock detection (wait chain traversal via `WCT` API)

## 5. Credential & Secret Management

**Risk Level:** LOW-MEDIUM

**Failure Mode:** API keys for OpenAI, Anthropic, Gemini, and GitHub tokens are stored on disk. Compromise of the settings file or DPAPI keys could leak credentials.

### Existing Mitigations

| Mitigation | Location | Effectiveness |
|------------|----------|---------------|
| DPAPI encryption for all API keys | `AppSettingsService.cs` ‚Äî `EncryptedApiKey`, `EncryptedAnthropicApiKey`, etc. | Strong ‚Äî tied to Windows user login |
| Runtime plaintext marked `[JsonIgnore]` | `AppSettingsService.cs:17-60` ‚Äî never serialized to disk | Strong ‚Äî keys only in memory |
| Per-provider key separation | `AppSettings` class ‚Äî separate encrypted fields per provider | Good ‚Äî limits blast radius |
| Environment variable fallback | README ‚Äî `OPENAI_API_KEY`, `ANTHROPIC_API_KEY` auto-detected | Moderate ‚Äî standard practice |
| API key validation before use | `ApiKeyValidatorTests.cs` ‚Äî empty, whitespace, format checks | Moderate |

### Gaps

- **No key rotation mechanism** ‚Äî keys are stored until manually changed
- **No secure key deletion on uninstall** ‚Äî `%LOCALAPPDATA%\CEAISuite\settings.json` persists
- **No memory protection** for runtime plaintext keys ‚Äî `string` fields are not pinned/zeroed
- **Gemini OAuth credentials** (client ID, client secret) also stored via DPAPI but refresh tokens have no expiry tracking

### Recommendations

1. Consider `SecureString` or pinned memory for runtime key storage (though .NET's SecureString is deprecated ‚Äî evaluate alternatives)
2. Add uninstall cleanup guidance or script
3. Track and warn on Gemini refresh token age

## 6. Platform & Architecture Risks

### Existing Mitigations

### Assessment

Well-documented and acknowledged. Not a concern for the current Windows-only v1.

**Risk Level:** LOW (acknowledged, user responsibility)

### Gaps

- No detection simulation tests

## 8. Cheat Table Serialization

**Risk Level:** LOW

**Failure Mode:** Loading/saving `.CT` cheat tables could corrupt data, lose pointer chains, or misinterpret data types.

### Existing Mitigations

- 10 comprehensive round-trip tests in `CheatTableRoundTripTests.cs`:
  - Simple entry address/type preservation
  - Script entry content preservation
  - Nested group hierarchy preservation
  - Pointer entry with offsets
  - Dropdown list values
  - ShowAsHex/ShowAsSigned flags
  - Lua script preservation
  - Empty CT handling
  - Mixed data types (Int32, Float, Double, String, ByteArray)
  - Entry count verification after round-trip
- `CrashRecoveryTests.cs` verifies export/import including corrupt file rejection

### Gaps

- **No malicious CT file test** ‚Äî what happens with XML entities, XXE attacks, extremely deep nesting?
- **No very large CT file test** ‚Äî performance with thousands of entries is untested

### Recommendation

Add a test with a crafted adversarial `.CT` file (XXE, entity expansion, >1000 entries).

## 9. Lua Scripting Engine

**Risk Level:** MEDIUM

**Failure Mode:** Lua scripting is partially implemented (~100 tests across 5 Lua test files) but represents approximately 15% of Cheat Engine's full Lua API surface. Scripts relying on CE-specific Lua functions may silently fail or behave unexpectedly.

### Existing Mitigations

- `LuaEngineTests.cs` (34 tests), `LuaCeApiTests.cs` (26 tests), `LuaGapClosureTests.cs` (13 tests), `LuaAutoAssemblerIntegrationTests.cs` (14 tests), `LuaConsoleViewModelTests.cs` (13 tests)
- CE API bindings are stub-tested for the implemented surface area

### Gaps

- **Sandbox is configured but not adversarially tested** ‚Äî `Preset_HardSandbox` disables `os`, `io`, `debug`, and dynamic code loading, and execution timeout is 30 seconds. However, no tests attempt sandbox escape (e.g., `debug.getinfo`, `loadstring`, FFI tricks)
- **Incomplete CE Lua API** ‚Äî users porting CE Lua scripts will hit missing functions (~15% of CE's Lua surface area)
- **No test for Lua memory limits** ‚Äî large string concatenation or table growth could exhaust host process memory within the 30-second timeout

### Recommendation

1. Add adversarial sandbox escape tests (attempt `os.execute`, `io.popen`, `debug.getinfo`, `loadstring`)
2. Add memory limit enforcement (MoonSharp supports memory tracking)
3. Document which CE Lua functions are supported vs. missing

## 10. Tool-to-UI Parity

**Risk Level:** LOW-MEDIUM

**Failure Mode:** Only ~21% of the ~90 AI tool functions have manual UI equivalents. Users relying on the UI without AI cannot access most capabilities.

### Assessment

This is a design decision, not a bug ‚Äî the philosophy is "AI assists, not replaces." However, if the AI provider is unavailable, users lose access to ~70 functions including pointer scanning, structure dissection, AOB signature generation, and session management.

### Recommendation

Prioritize UI for critical-path functions: pointer scanner, memory scanner refinement, and session save/load.
